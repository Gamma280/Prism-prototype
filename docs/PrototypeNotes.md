# Prism Prototype — Known Limitations & Design Decisions

This document explains what is intentionally incomplete, what is a placeholder, and why certain decisions were made for the prototype. These are not bugs — they are deliberate scope choices to keep the demo focused on demonstrating the core Prism concept.

---

## 1. Projects Section — Not Implemented

**Current state:** Clicking "Projects" in the sidebar does nothing. The nav item exists as a visual placeholder only.

**What it would do in a full product:**
Projects would let an analyst group related research chats together — for example, all conversations about a single company (e.g. Tata Consumer FY25 coverage) or a single research report could be organised under one project. Each project would have its own verified claims history, a summary of all flagged figures, and a completion status showing how many claims have been verified across the project.

**Why it was skipped for the prototype:**
The prototype's goal is to demonstrate the core Prism loop: Prompt → AI Response → Claim Flagging → Verification. Projects is a organisational feature that sits on top of that loop and does not change the core demonstration. It would require a more complex data model (projects containing sessions containing messages) and a dedicated project view — scope that adds time without adding to the proof of concept.

**Priority for production:** Medium — important for workflow, not for the initial demo.

---

## 2. Prism Settings Section — Not Implemented

**Current state:** Clicking "Prism Settings" opens the onboarding modal (reuses the help button behaviour). There is no actual settings panel.

**What it would contain in a full product:**
- **Profile settings:** Change the display name (currently hardcoded as "Guest"), profile picture, and analyst tier (Junior / Senior)
- **Senior Analyst Mode:** A toggle to hide all Prism markers for experienced users who want a clean reading experience — mentioned in the problem statement as an optional enhancement
- **Notification preferences:** Whether to show the "All claims checked" banner, whether to auto-collapse the Prism Assumes panel
- **API / model configuration:** In production, this would let an organisation configure their own API key or choose which LLM provider to use
- **Data & privacy:** Clear stored chat history, export verified claims, manage localStorage data

**Why it was skipped for the prototype:**
Settings require UI for every setting plus persistence logic for each preference. None of these settings change what a demo audience needs to see. The demonstration value of Prism is in the flagging, verification, and claims workflow — not in configuration.

**Priority for production:** Medium — needed before any multi-user rollout.

## 3. Output Accuracy — Illustrative, Not Verified

**Current state:** The responses generated by the AI (Groq / LLaMA 3.3 70B) contain real financial figures, historical data, and market analysis drawn from the model's training data. However, **this data has not been independently verified and may be outdated, inaccurate, or hallucinated.**

**This is intentional and is actually central to the Prism concept:**

The prototype exists precisely to show what happens when an analyst receives a polished, confident-looking AI response — and why they need a layer like Prism to flag what requires verification before submission. The AI output is not the product. Prism's claim detection layer on top of that output is the product.

Specific known limitations of the AI output:
- **Training cutoff:** LLaMA 3.3's training data has a knowledge cutoff. Any figures described as "recent" or "current" may be 6–18 months out of date
- **Hallucination risk:** The model may cite specific percentage figures (e.g. "EBITDA margin of 14.2% in FY25") that appear precise but have not been cross-checked against BSE filings or annual reports
- **Indian market specificity:** While the system prompt instructs the model to focus on Indian markets, BSE, RBI, and SEBI, the model's knowledge of specific Indian company financials varies in reliability
- **No real-time data:** The model has no access to live stock prices, current commodity prices, or latest quarterly results

**What this means for the demo:**
When showing the prototype, the red and yellow flag markers on the AI response are demonstrating exactly the right behaviour — they are correctly identifying that these figures need to be verified externally. The Prism layer is working as designed even if the underlying figure turns out to be wrong.

**For a production version:** The system prompt engineering, fact-checking pipeline, and retrieval-augmented generation (RAG) from live BSE/NSE/RBI data feeds would replace the current approach.

## 4. Claude Branding with Groq Backend — Intentional Mismatch

**Current state:** The interface displays the Claude logo (Claudelogo.png), uses the name "Claude" in the sidebar brand, and is styled to resemble Claude's interface. However, the AI responses are generated by **Groq's inference API running LLaMA 3.3 70B** — a Meta open-source model, not Anthropic's Claude.

**Why this decision was made:**
The problem statement explicitly describes Prism as "a claim-level evaluation layer for Claude." The target users — junior financial analysts — use Claude daily in their workflows. The prototype is designed to look and feel exactly like the tool they already use, so that the demo is immediately recognisable and the value of the Prism layer is clear in context.

Using Claude as the visual reference makes the prototype more persuasive and easier to understand for the intended audience. The Claude UI is the design benchmark the problem statement specifies.

Groq was chosen as the AI backend because:
- It is free to use with a straightforward API key
- The LLaMA 3.3 70B model produces strong financial analysis for a prototype
- No usage costs during development and demos
- Fast inference speed (~200 tokens/second) makes the demo feel responsive

**For a production version:** The backend would use the actual Claude API (claude-sonnet-4-x or claude-opus-4-x via Anthropic). The system prompt is already written to work with any capable LLM and would transfer directly to Claude with minimal changes.

**Disclosure for demos:** When presenting the prototype, it is accurate to say "this is a Claude-like interface" or "this is what Prism would look like built on top of Claude" — the interface design is faithful to Claude, even if the current AI backend is different.

## 5. Browser localStorage — Temporary Memory, Not Production Storage

**Current state:** All chat sessions, verified claims, starred conversations, and message history are stored in the browser's `localStorage`. This is the built-in key-value storage that every browser provides, limited to a single device and a single browser.

**What this means in practice:**
- If a user opens the prototype in a different browser (e.g. Chrome vs Firefox), they will see an empty session list — the data does not transfer between browsers
- If a user opens the prototype on a different device (e.g. their phone vs their laptop), the same — each device has its own separate storage
- If a user clears their browser cache or "Clear site data", all chat history and verified claims will be deleted
- The storage limit is typically 5–10MB per origin, which is sufficient for hundreds of chat sessions but will eventually fill up without a cleanup mechanism
- There is no user account, no login, no server — the data never leaves the user's browser

**Why localStorage was used for the prototype:**
localStorage requires zero infrastructure — no database, no backend, no login system. For a single-page prototype that needs to demonstrate the Prism workflow without requiring a server, it is the correct choice. It also means the prototype is fully deployable as a static site on Vercel with no running costs.

**What production storage would look like:**
In a production version, each analyst would have a user account. All sessions, messages, and verified claim states would be stored in a server-side database (e.g. Vercel KV, PlanetScale, Supabase, or Firebase). The `persistSave()` and `persistLoad()` functions in the codebase are the only two places that need to be updated — the rest of the application logic is storage-agnostic. The switch from localStorage to a database API is a drop-in replacement at those two functions.

Additionally, production storage would unlock cross-device sync, team features (sharing verified claims with a senior analyst), and audit trails (who verified which claim and when).
